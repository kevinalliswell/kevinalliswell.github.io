# Kevin Blog 写作模板

> 目标：把复杂技术或抽象思考，整理成普通读者也能读懂、并且能直接拿去用的判断型长文。

---

## 一、风格定位

这套模板适合写：
- 技术教程
- 工具指南
- 项目复盘
- 趋势分析
- 方法论文章

不追求论文腔，也不追求营销号口吻。
核心特点是：
1. **结论先行**
2. **结构清晰**
3. **概念讲人话**
4. **案例和决策并重**
5. **既讲是什么，也讲为什么**

---

## 二、推荐标题公式

从下面 4 类里选一种：

### 1. 教程指南型
- 一文搞懂 X：从入门到精通
- X 完全使用指南：从零开始到实战
- X 最全教程：原理、配置与最佳实践

### 2. 趋势判断型
- X 不再是重点：Y 才是下一阶段的关键
- 从 X 到 Y：一场正在发生的范式转移
- 为什么说 X 正在重塑 Y

### 3. 实战复盘型
- 从 A 到 B：我用 X 完成 Y 的全过程
- 我与 AI 协作完成 X 的完整复盘
- 一次真实项目实战：如何从 0 到 1 做出 X

### 4. 思考方法型
- 为什么说 X 不是一个技巧，而是一套系统
- 真正重要的不是 X，而是 Y
- 一套可以复用的 X 决策框架

---

## 三、标准文章结构

### 1. Frontmatter

```yaml
---
title: "文章标题"
date: 2026-04-17T21:00:00+08:00
draft: false
tags: ["标签1", "标签2", "标签3"]
categories: ["技术"]
author: "Kevin"
description: "用一句话概括文章价值"
---
```

### 2. 开头摘要（blockquote）

用 1 句话说清本文价值。

示例：

> 很多人写技术文章，问题不在于懂得不够多，而在于没有把“自己知道的东西”翻译成“别人看得懂、用得上”的结构。

---

### 3. 第一部分：先讲结论 / 一句话定义

开篇不要绕。
直接回答：
- 这篇文章在讲什么
- 你的核心判断是什么
- 读者为什么值得继续看

可用句式：
- **先说结论：**
- **一句话定义：**
- **我的判断是：**
- **如果只记住一件事，那就是：**

---

### 4. 第二部分：问题背景 / 为什么值得讨论

说明：
- 这个问题为什么出现
- 读者常见误区是什么
- 你为什么觉得它重要

目标：把“为什么要读”讲清楚。

---

### 5. 第三部分：核心概念拆解

如果有抽象概念，必须翻译成人话。

推荐写法：
1. 先给正式定义
2. 再给一句通俗解释
3. 再给一个类比或场景

模板：

```md
## 什么是 X？

### 一句话定义

**X 是……**

### 人话解释

换句话说，X 的本质是……

### 类比理解

可以把它理解成……
```

---

### 6. 第四部分：展开论证 / 分点拆解

通常拆成 3-5 个部分最合适。
每一部分都尽量满足：
- 一个清晰小标题
- 一个明确观点
- 一段解释
- 一个例子 / 对比 / 数据 / 场景

推荐句式：
- 这件事真正关键的地方在于……
- 问题不只是……而是……
- 表面上看是 X，本质上其实是 Y
- 这也是为什么……

---

### 7. 第五部分：案例 / 实战 / 决策过程

这是 Kevin 风格里辨识度最高的一部分。

如果是技术或项目文章，建议固定回答这几个问题：
1. **输入是什么**
2. **目标是什么**
3. **有哪些方案**
4. **为什么选这个方案**
5. **AI / 工具做了什么**
6. **人做了什么判断**
7. **哪里最容易出错**

推荐小标题：
- 背景
- 输入
- 方案选择
- 我做的决策
- 反思

---

### 8. 第六部分：风险、边界与反例

避免文章显得过满、过绝对。

可写：
- 这套方法适用什么场景
- 不适合什么场景
- 容易踩的坑
- 哪些地方仍然需要人工判断

推荐句式：
- 这里要特别警惕的是……
- 这并不意味着……
- 这套方法的边界在于……
- 真正需要人把关的，是……

---

### 9. 第七部分：总结 / 可执行清单

结尾不要空泛抒情，尽量给读者可带走的东西。

推荐两种结尾方式：

#### 方式 A：三点总结
```md
## 三点总结

1. X 是……
2. 真正关键的是……
3. 如果你要开始，先做……
```

#### 方式 B：行动清单
```md
## 如果你想马上开始，可以这样做

1. 先……
2. 再……
3. 最后……
```

---

## 四、语言风格要求

### 应该多用
- 短句
- 判断句
- 对比句
- 结构化列表
- 表格
- blockquote 金句
- “换句话说”“本质上”“这也是为什么” 这类过渡语

### 应该少用
- 空泛形容词堆砌
- 过度学术化表达
- 没有场景支撑的抽象结论
- 很长很绕的段落

---

## 五、Kevin 风格的固定加分项

### 1. 结论先行
不要把结论藏到后面。

### 2. 把概念翻译成人话
每出现一个新概念，都问自己一句：
**“普通读者第一次看到，会不会卡住？”**

### 3. 不只讲做法，还讲决策依据
比“怎么做”更重要的是“为什么这样做”。

### 4. 允许有判断，但不要装腔
语气可以坚定，但不要故作高深。

### 5. 让读者读完能带走一个框架
最好的文章不是“信息很多”，而是“读完以后脑子里多了一个可复用的结构”。

## 七、发文前自检清单

发布前检查：

- [ ] 标题是否有明确承诺
- [ ] 开头 3 段内是否给出核心判断
- [ ] 每个抽象概念是否讲了“人话版本”
- [ ] 是否有至少 1 个案例 / 场景 / 对比
- [ ] 是否解释了“为什么这样做”
- [ ] 是否写了边界和风险
- [ ] 结尾是否给出可执行总结
- [ ] description 是否可直接用于 SEO 摘要

---

## 八、一句话总纲

> **把复杂问题写简单，把个人经验写成方法，把单次实践写成可复用框架。**
