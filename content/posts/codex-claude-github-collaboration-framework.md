> 单个 AI 一干到底，最后收获一堆难以验收的代码。真正可控的做法是：拆阶段、设关卡、让两个 AI 互为补位，人在关键节点把关。

---

## 先说结论

这是一套**双 AI + 中枢仓库**的协作开发框架，核心分工是：

- **Codex** —— 规划者 + 建造者：制定目标、初始化项目、生成规则文档，并执行每个阶段的开发
- **Claude Code** —— 验收者 + 编排者：审查每个阶段的进度和质量，生成下一阶段的开发指令
- **GitHub** —— 同步中枢：所有交付物的唯一真相源，承载分支管理和阶段流转

如果只记住一件事：**不要让 AI 一口气把项目做完。把任务拆成 Phase A/B/C/D，每个阶段做完就停下来验收，通过了再放行下一阶段。** 阶段化 + 验收关卡，是把"AI 写代码"变成"AI 可控交付"的关键。

---

## 背景：为什么需要这样一套框架

让 AI 写代码已经不是问题，问题是**怎么让它写出可控、可验收、可持续推进的代码**。

直接把一个大需求丢给 AI，常见的失败模式有三种：

1. **一干到底，无法验收** —— AI 闷头写了几千行，等你回来看，方向已经偏了，返工成本极高
2. **上下文丢失，前后矛盾** —— 单次会话塞太多任务，后面的实现忘了前面的约定
3. **分支混乱，改动难追溯** —— 全在一个分支上反复横跳，出了问题无法回滚到稳定状态

这套框架用三个机制解决这三个问题：

| 失败模式 | 对应机制 |
|---------|---------|
| 一干到底无法验收 | **分阶段 + 阶段间人工/AI 验收** |
| 上下文丢失 | **规则文档固化约定 + 每阶段独立指令** |
| 分支混乱 | **main / dev / feature 三层分支管理** |

## 核心概念：三个角色的分工

### 一句话定义

**这是一套"Codex 造、Claude Code 验、GitHub 同步"的流水线**，AI 负责执行，人负责在关卡处做判断。

### 人话解释

把它想象成一个微型工程团队：

- Codex 是**干活的工程师**——既做前期设计，也写代码
- Claude Code 是**技术主管 + QA**——检查工程师的产出，决定下一步做什么、怎么做
- GitHub 是**项目看板 + 代码库**——所有人围着它协作，状态以它为准
- 你是**项目负责人**——在每个阶段交付时拍板：过，还是返工

### 为什么用两个 AI 而不是一个

关键在于**视角分离**。

让同一个 AI 既当运动员又当裁判，它很难发现自己的问题——它会倾向于认为自己写的代码是对的。**换一个 AI 来验收，相当于引入了独立的第二双眼睛。** Claude Code 在审查 Codex 产出时，不背负"这是我写的"的包袱，更容易发现遗漏和偏差。

这和代码评审里"作者不评审自己的 PR"是同一个道理。

## 完整工作流

整体流程是一个**循环**：规划 → 初始化 → 分阶段开发 → 验收 → 下发下一阶段 → 再开发……直到达标。

```
Codex 制定目标 + 初始化项目 + 生成规则文档
      ↓
提交到远端仓库（GitHub）
      ↓
编写 Codex 开发指令（含远端仓库地址）
      ↓
Codex 拉取项目 → 本地初始化 → goal 模式开发 Phase A
      ↓
人工验收 Phase A（优化 or 直接过）→ 提交推送远端
      ↓
Claude Code 验收 Phase A 的进度与质量 → 生成 Phase B 开发指令
      ↓
Phase B 指令交给 Codex → 实现 Phase B
      ↓
   （循环往复，直到项目达标）
```

下面逐步拆解。

### 步骤一：用 Codex 做全局规划与初始化

让 Codex 先不要写业务代码，而是做两件事：**定目标**和**搭骨架**。

**定目标**：让 Codex 把项目的整体开发目标、技术选型、阶段划分想清楚。

**搭骨架**：生成项目初始化结构和一整套规则文档：

| 文档 | 作用 |
|------|------|
| `AGENTS.md` | 给 Codex 的项目规则：编码规范、约束、命令 |
| `CLAUDE.md` | 给 Claude Code 的项目规则：同上，供验收方理解项目 |
| `ARCHITECTURE.md` | 架构设计：模块划分、数据流、关键决策 |
| `TASKS.md` | 任务清单，**按阶段拆分**（Phase A/B/C/D...） |
| GitHub 初始文件 | `README.md`、`.gitignore`、`LICENSE`、Issue 模板等 |

`TASKS.md` 是整套框架的"作战地图"。它把项目拆成有先后依赖的阶段，每个阶段有明确的交付物和验收标准。例如：

```md
## Phase A：基础设施
- [ ] 项目脚手架与依赖
- [ ] 数据模型定义
- [ ] CI/CD 基础配置
验收标准：能跑起来，测试框架就位

## Phase B：核心业务逻辑
- [ ] 模块 X 实现
- [ ] 模块 Y 实现
验收标准：核心流程跑通，单测覆盖关键路径

## Phase C：接口与集成
...

## Phase D：打磨与上线准备
...
```

### 步骤二：提交到远端仓库

把初始化的项目和规则文档提交到 GitHub。**从这一刻起，GitHub 就是唯一真相源。** 后续两个 AI 都围绕这个远端仓库协作——Codex 推代码上去，Claude Code 拉代码下来验收。

### 步骤三：编写 Codex 开发指令

为每个阶段编写一段开发指令提示词。**关键：提示词里要包含远端仓库地址**，让 Codex 知道从哪里拉项目。

一个 Phase A 指令的骨架：

```md
仓库：https://github.com/your-name/your-project

任务：实现 Phase A（见仓库内 TASKS.md）

要求：
1. 从远端仓库拉取最新代码到本地
2. 遵循 AGENTS.md 的编码规范和 ARCHITECTURE.md 的架构
3. 只做 Phase A 范围内的任务，不要提前做 Phase B
4. 在专用 feature 分支上开发，不要直接动 main/dev
5. 完成后自测，确保验收标准达成
```

第 3 条尤其重要——**明确边界，禁止 AI 越界提前开发后续阶段**。这是保证"分阶段"不流于形式的关键约束。

### 步骤四：Codex 拉起项目，goal 模式开发

把指令交给 Codex，它从远端拉取项目、在本地完成初始化，然后启动 **goal 模式**，自主完成 Phase A 的任务。

goal 模式适合阶段内的连续开发——目标明确、边界清晰，让 Codex 在这个范围内自主推进。

### 步骤五：人工验收 Phase A

Phase A 完成后，**停下来**。这是第一道关卡。

人工过一遍：
- 验收标准达成了吗？
- 代码质量能接受吗？
- 有没有埋下后续阶段的坑？

然后做决定：
- **需要优化** → 让 Codex 改，改完再验
- **直接过** → 提交推送到远端仓库

无论哪种，结果都要落到 GitHub 上，保证远端是最新的、验收过的状态。

### 步骤六：Claude Code 交叉验收 + 下发下一阶段

这是双 AI 框架的精髓。让 **Claude Code 独立验收 Phase A** 的进度和质量——它会拉取远端最新代码，对照 `TASKS.md` 和 `ARCHITECTURE.md` 检查：

- Phase A 的任务是否真的全部完成
- 实现是否符合架构约定
- 有没有 Codex 自己没发现的问题

验收通过后，**让 Claude Code 生成 Phase B 的 Codex 开发指令**。因为 Claude Code 刚刚深入理解了 Phase A 的实际状态，它生成的 Phase B 指令会更贴合当前代码的真实情况，而不是凭空想象。

### 步骤七：循环

把 Phase B 指令交给 Codex → 实现 → 人工验收 → Claude Code 验收并生成 Phase C 指令……

**如此循环往复，直到项目达标。**

## 关键设计一：分阶段 + 阶段间验证

> 任务要分阶段拆解，中间插入验证，不要一干到底。

这是整套框架最重要的原则。

**为什么不能一干到底？** 因为 AI 的错误会**累积放大**。如果 Phase A 埋了一个架构问题，没人发现，Phase B、C、D 全部建立在错误的地基上，等到最后验收时，返工成本是指数级的。

**阶段间验证的本质，是把"一次大验收"拆成"多次小验收"。** 每个阶段都是一道关卡，问题在它产生的阶段就被拦截，不会流到下游。

| 一干到底 | 分阶段验证 |
|---------|-----------|
| 错误累积到最后才暴露 | 错误在当前阶段就拦截 |
| 返工成本指数级 | 返工成本线性、可控 |
| 验收时面对几千行代码 | 每次只验收一个阶段 |
| AI 上下文越来越乱 | 每阶段上下文清晰 |

## 关键设计二：分支与工作树的管理

> 区分分支（branch）和工作树（worktree），两者不要混，做好分支管理。

### 分支 vs 工作树

先把概念讲清楚：

- **分支（branch）**：同一个工作目录下，通过 `git checkout` 切换不同的开发线
- **工作树（worktree）**：`git worktree` 允许你把不同分支**同时**检出到不同的物理目录，互不干扰

**为什么要区分？** 当两个 AI 同时在动代码时，如果共用一个工作目录来回切分支，极易互相覆盖、污染对方的未提交改动。用 worktree 给不同的工作（比如 Codex 开发 vs Claude Code 验收）分配独立目录，物理隔离，互不打架。

```bash
# 主目录在 dev 分支
git worktree add ../project-phase-b feature/phase-b   # Codex 在这个独立目录开发 Phase B
git worktree add ../project-review   dev               # Claude Code 在另一个目录验收
```

**原则：开发和验收用不同的工作树，不要在同一个目录里反复切分支。**

### 三层分支模型

> 保持三个主要分支：main（完成第一版后锁死）、dev、开发中的分支，逐项往前合并。

```
feature/phase-x   （开发中的分支，每个阶段一条）
      ↓ 阶段验收通过后合并
    dev           （集成分支，持续推进）
      ↓ 第一版完成后合并并锁死
    main          （稳定版，锁死保护）
```

| 分支 | 角色 | 规则 |
|------|------|------|
| `main` | 稳定版 | 完成第一版后**锁死**，开启分支保护，禁止直接 push |
| `dev` | 集成分支 | 各阶段验收通过后合并到这里，始终保持可运行 |
| `feature/phase-x` | 阶段开发分支 | 每个 Phase 一条，开发完合并到 dev 后即可删除 |

**逐项往前合并**的含义：改动从 `feature` → `dev` → `main` 单向流动，每一步都经过验收。永远不在 main 上直接开发，保证 main 任何时候都是可信赖的稳定状态。

## 风险与边界

### 1. 规则文档会漂移

`AGENTS.md`、`ARCHITECTURE.md` 这些文档是 AI 的行为准则，但随着开发推进，实现可能偏离文档。**每个阶段验收时，要顺带检查文档和代码是否还一致**，必要时更新文档。否则后期 AI 会基于过时的文档做出错误决策。

### 2. 阶段拆分的粒度需要拿捏

阶段太大，失去了"中间验证"的意义；阶段太小，验收开销淹没了开发收益。**经验值：每个阶段控制在能独立验收、且半天到两天能完成的范围。**

### 3. 两个 AI 的规则要对齐

`AGENTS.md`（给 Codex）和 `CLAUDE.md`（给 Claude Code）描述的应该是**同一套项目约定**。如果两份文档冲突，会出现"Codex 按 A 标准写、Claude Code 按 B 标准验"的荒诞局面。保持两者核心约定一致。

### 4. 人不能完全退出

这套框架降低了人的工作量，但**没有取消人的判断**。阶段验收、方向把关、要不要进下一阶段——这些决策仍然需要人来做。AI 负责执行和交叉检查，人负责对最终结果负责。

### 5. 成本与节奏

双 AI + 多阶段意味着更多的 token 消耗和流程开销。**这套框架适合中大型、有清晰阶段划分的项目；对于一个几十行的脚本，直接写比走流程更快。**

## 如果你想马上开始，可以这样做

1. **先让 Codex 做规划**：把需求丢给 Codex，让它产出 `TASKS.md`（分 Phase）+ `ARCHITECTURE.md` + `AGENTS.md` + `CLAUDE.md`，而不是直接写代码
2. **建好仓库和三层分支**：初始化推到 GitHub，建好 `main` / `dev`，为 Phase A 开一条 `feature/phase-a`
3. **写第一段 Codex 指令**：含仓库地址、阶段范围、分支约束，交给 Codex 在独立 worktree 里跑 Phase A
4. **设好第一道关卡**：Phase A 完成后人工验收 → 推送远端 → 让 Claude Code 交叉验收并生成 Phase B 指令
5. **循环到达标**：每个阶段都走"开发 → 双重验收 → 合并 → 下发"，逐项往前合并，直到 main 锁死第一版

---

## 三点总结

1. **双 AI 分工，视角分离** —— Codex 造、Claude Code 验、GitHub 同步。换一个 AI 验收，相当于引入独立的第二双眼睛
2. **分阶段 + 阶段间验证** —— 不要一干到底。把一次大验收拆成多次小验收，错误在产生的阶段就被拦截
3. **三层分支 + 工作树隔离** —— main 锁死、dev 集成、feature 开发，逐项往前合并；开发和验收用不同 worktree，物理隔离不打架

最后：

> **AI 协作开发的可控性，不来自更强的模型，而来自更好的流程。拆阶段、设关卡、分分支——把不确定性关进结构里，AI 才能真正成为可靠的生产力。**
